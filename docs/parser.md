# The Argon parser

This document describes the hand-written parser that turns Argon source text
into an abstract syntax tree (AST). It is a reference for anyone modifying the
parser, the lexer, or the grammar.

The parser lives in `core/compiler/src/parser/`:

| File         | Responsibility |
|--------------|----------------|
| `token.rs`   | `TokenKind` (the lexical categories), `Token` (a kind + byte span), and the keyword classifier. |
| `lexer.rs`   | `Lexer` — a streaming byte lexer that produces one `Token` at a time. |
| `grammar.rs` | `Parser` — a recursive-descent + Pratt parser that builds the AST. Most of the logic lives here. |
| `mod.rs`     | The public entry points (`parse_ast`, `parse_cell`), the `ParseError` type, and the `#[cfg(test)]` test suite. |

The AST node definitions are in `core/compiler/src/ast/mod.rs`. A descriptive
(non-executable) reference grammar is kept in sync at
`core/compiler/grammar/Argon.g4`.

---

## 1. Overview

The parser was originally generated by [ANTLR](https://www.antlr.org/). It has
been replaced by a hand-written parser; the ANTLR tool, its generated code, and
the `antlr-rust` runtime dependency are gone, which also removes the Java/Maven
build step. The hand-written parser is:

- **Single-pass.** There is no intermediate concrete syntax tree (CST). The
  token stream is walked once and AST nodes are constructed directly.
- **Zero-copy.** Tokens carry only a kind and a byte range. Identifier and
  string text is borrowed straight out of the source `&str` by slicing that
  range, so lexing and parsing allocate nothing for token text.
- **Recursive descent + Pratt.** Declarations, statements, and types use plain
  recursive descent (one function per grammar rule). Expressions use a
  [Pratt](https://en.wikipedia.org/wiki/Operator-precedence_parser)
  (precedence-climbing) loop so operator precedence and associativity live in a
  single table instead of a tower of grammar rules.
- **Error-recovering and panic-free.** A parse never aborts on the first error
  and never panics on malformed input. It accumulates every diagnostic it can
  and always returns a (possibly degraded) AST. This matters for the language
  server, which re-parses incomplete files on every keystroke.

The high-level pipeline:

```
                  trim leading                bytes + base offset
  ArcStr / &str  ─ whitespace ─▶  &str  ──────────────────────────▶  Lexer
   (raw source)   (offset_base)  (trimmed)                             │
                                                                       │ next_token()
                                                                       ▼
                                                                    Token stream
                                                                       │
                                                          two-token window (cur, nxt)
                                                                       ▼
                                                                    Parser
                                                            (recursive descent + Pratt)
                                                                       │
                                                                       ▼
                                                       Ast<&str, ParseMetadata>  +  Vec<ParseError>
```

## 2. Entry points and the offset model

Two public functions in `mod.rs` wrap the parser:

```rust
pub fn parse_ast(input: ArcStr, path: PathBuf)
    -> Result<AnnotatedParseAst, Vec<ParseError>>;

pub fn parse_cell(input: &str)
    -> Result<CallExpr<&str, ParseMetadata>, Vec<ParseError>>;
```

`parse_ast` parses a whole source file (`ast : decl* EOF`). On success it runs
the *annotation pass* (`ast::annotated::AnnotatedAst::new`), which re-slices
every identifier and string value out of the shared `ArcStr` by its span. On any
syntax error it returns every collected diagnostic.

`parse_cell` parses a single cell invocation, e.g. `top(1., 5)` — see
[§11](#11-the-parse_cell-contract).

### Leading-whitespace trimming and `offset_base`

Both entry points do the same preprocessing before constructing the parser:

```rust
let normalized = input.trim_start_matches(char::is_whitespace);
let offset_base = input.len() - normalized.len();
let mut parser = grammar::Parser::new(normalized, offset_base);
```

The lexer scans the *trimmed* buffer, but every span the parser produces must
index the *original* (untrimmed) input, because the annotation pass re-slices
the original `ArcStr`. To bridge the two, the lexer adds `offset_base` (the
number of bytes trimmed) to every offset it emits. The result: byte offsets in
the AST are always relative to the original input.

> **Known leniency:** the trim uses `char::is_whitespace` (Unicode-aware) while
> the lexer's `skip_trivia` only skips ASCII ` \t\r\n`. A leading non-ASCII
> whitespace byte (NBSP, BOM, …) is therefore folded into `offset_base` and the
> input parses, where the old ANTLR `WS` rule would have rejected it. This is
> harmless and currently accepted.

### Byte-exact spans

Spans must be **byte-exact** because the annotation pass indexes the source with
them. Two consequences run through the whole implementation:

- `cfgrammar::Span::new(start, end)` **panics if `end < start`.** Any code that
  builds a span from two independently-derived offsets must guarantee
  `start <= end` (see [`finish_span`](#56-spans-finish_span-and-slicing)).
- Token boundaries always land on UTF-8 char boundaries (the lexer guarantees
  this — see [§4](#4-the-lexer)), so slicing a token's span out of the source is
  always valid UTF-8.

## 3. Tokens (`token.rs`)

A `Token` is deliberately tiny — a kind plus a half-open byte range. It owns no
text:

```rust
pub struct Token {
    pub kind: TokenKind,
    pub start: u32,   // inclusive, original coords (already includes offset_base)
    pub end: u32,     // exclusive
}
```

Using `u32` offsets keeps tokens small. (This caps source files at 4 GiB, which
is irrelevant for Argon.)

`TokenKind` is a flat `#[repr(u8)]` enum grouping:

- **Keywords** — `enum struct match const cell mod if fn else let for in as
  true false` (15 of them).
- **Names & literals** — `Ident`, `Annotation` (`#name`), `IntLit`, `StrLit`.
- **Multi-character operators** — `:: => == != >= <= ->`.
- **Single-character operators / punctuation** — `< > = ! + - * / % ( ) { } [ ]
  . : ; ,`.
- **`Eof`** — the empty range `[len, len)`.
- **`Error`** — a byte that begins no valid token, or an unterminated string. It
  carries a span so a diagnostic can point at it; the parser turns it into a
  `ParseError` and recovers.

`TokenKind::describe()` returns the human-readable name used in error messages
(e.g. `')'`, `identifier`, `end of input`).

### Keyword classification

`keyword_or_ident(&[u8]) -> TokenKind` decides whether a freshly scanned
identifier slice is a keyword. Because there are only 15 short, disjoint
keywords, it matches on `(length, bytes)`:

```rust
match s.len() {
    2 => match s { b"fn" => KwFn, b"if" => KwIf, /* … */ _ => Ident },
    3 => match s { b"let" => KwLet, /* … */ _ => Ident },
    // …
    _ => Ident,
}
```

This lowers to a length jump table plus a `memcmp` and beats a hash map with
zero setup cost.

## 4. The lexer (`lexer.rs`)

`Lexer` scans the source as raw bytes (`&[u8]`) and produces one token per call
to `next_token()`. It holds only three fields: the byte slice, the current
position `pos`, and the `base` offset added to every emitted span.

### Why bytes are safe

All token *boundaries* fall on ASCII bytes: identifiers, keywords, numbers, and
operators are entirely ASCII, and non-ASCII bytes can only appear *inside*
string or comment bodies (which are scanned byte-by-byte and never sliced
mid-character). So every offset the lexer emits is a valid UTF-8 char boundary.

### Trivia

`skip_trivia` is called at the start of every `next_token`. It skips ASCII
whitespace and `//` line comments (everything to the next `\n`/`\r`). Comments
and whitespace are *trivia* — they never become tokens. (A comment before the
first declaration therefore parses fine, which ANTLR's grammar rejected.)

### Dispatch

After skipping trivia, `next_token` dispatches on the first byte:

| First byte            | Rule                       | Token kind(s) |
|-----------------------|----------------------------|---------------|
| `_` or ASCII letter   | `lex_ident_or_keyword`     | `Ident` or a keyword |
| `#`                   | `lex_annotation`           | `Annotation` (or `Error` for a lone `#`) |
| `0`–`9`               | `lex_int`                  | `IntLit` |
| `"`                   | `lex_string`               | `StrLit` (or `Error` if unterminated) |
| anything else         | `lex_operator`             | an operator, or `Error` |
| past end of input     | —                          | `Eof` |

Notes on individual rules:

- **`lex_annotation`** requires the `#` to be followed by an identifier start;
  otherwise it emits a one-byte `Error`. The `#` is part of the token span; the
  parser strips it later.
- **`lex_int`** scans digits only. The fractional part of a float (`.5`) is *not*
  lexed here — the parser assembles floats from `IntLit DOT IntLit?` tokens (see
  [§9.3](#93-integers-vs-floats)).
- **`lex_string`** scans `"…"` with no escapes; a newline or EOF before the
  closing quote produces an `Error` token spanning what was consumed.
- **`lex_operator`** uses **maximal munch**: two-character operators are matched
  before their one-character prefixes (`::` before `:`, `==`/`=>` before `=`,
  `->` before `-`, etc.).

### Error tokens and UTF-8

When `lex_operator` hits a byte that begins no operator (e.g. `@`, or a
multi-byte character like `€`), it emits a single `Error` token spanning the
whole UTF-8 character and advances to the next char boundary:

```rust
let mut end = start + 1;
while end < n && self.src[end] & 0xC0 == 0x80 {   // skip continuation bytes
    end += 1;
}
```

`0b10xx_xxxx` is the UTF-8 continuation-byte pattern — the same rule as
`str::is_char_boundary`. Advancing to the boundary keeps every subsequent span
char-aligned, so the parser never slices mid-character.

## 5. The parser core (`grammar.rs`)

### 5.1 State

```rust
pub struct Parser<'a> {
    src: &'a str,        // the trimmed buffer
    base: usize,         // offset_base; subtract to index `src`
    lexer: Lexer<'a>,
    cur: Token,          // the token to act on
    nxt: Token,          // one token of lookahead
    prev_end: u32,       // end offset of the most recently consumed token
    ntok: u64,           // monotonic consumed-token count (loop progress guard)
    depth: u32,          // recursion-depth guard
    pub errors: Vec<ParseError>,
}
```

### 5.2 Two-token lookahead

The parser keeps a sliding window of `cur` and `nxt`. `Parser::new` primes it by
lexing two tokens. One token of lookahead is exactly what the grammar needs —
the only place it's used is `is_kwarg_start`, which distinguishes a keyword
argument `name = expr` from a positional one by peeking past the identifier at
the `=`:

```rust
fn is_kwarg_start(&self) -> bool {
    self.cur.kind == TokenKind::Ident && self.nxt.kind == TokenKind::Eq
}
```

### 5.3 Token plumbing

Three primitives drive everything:

- **`bump() -> Token`** — consume `cur`, shift `nxt` in, lex a new `nxt`, record
  `prev_end`, and increment `ntok`. Returns the consumed token.
- **`eat(k) -> bool`** — `bump` if `cur` is of kind `k`, else do nothing.
  Returns whether it matched. Used for optional tokens.
- **`expect(k) -> Token`** — `bump` if `cur` is `k`; otherwise record an
  "expected …, found …" error and return a **zero-width synthetic token** at the
  current position *without consuming*. Callers can still derive a (degraded)
  span from the synthetic token, and the progress guards (§10) prevent the
  non-consumption from stalling a loop.

### 5.4 Recursion-depth guard

`parse_expr`, `parse_unannotated_scope`, and `parse_ty_spec` are mutually
recursive and can be driven arbitrarily deep by pathological input (`[[[[…`,
`((((…`, deeply nested `if`). `enter_depth`/`exit_depth` bump a shared counter
and bail (recording a "nesting too deep" error and returning a degraded node)
once it exceeds `MAX_DEPTH` (256), which protects the native stack:

```rust
fn enter_depth(&mut self) -> bool {
    self.depth += 1;
    if self.depth > MAX_DEPTH { self.depth -= 1; false } else { true }
}
```

> The counter is shared and slightly over-counts (one source nesting level can
> pass through both `parse_expr` and `parse_unannotated_scope`), so the
> effective limit on user nesting is roughly half of `MAX_DEPTH`. Real Argon
> nests fewer than ten deep, so this is academic; the guard exists only to turn
> a stack overflow into a clean diagnostic.

### 5.5 Spans: `finish_span` and slicing

Leaf nodes (a single token) span exactly that token: `span(t)`. Composite nodes
span from the first token's start to the end of the last token they cover. The
"end of the last token" is `prev_end`, and composite spans are closed with:

```rust
fn finish_span(&self, lo: u32) -> Span {
    Span::new(lo as usize, self.prev_end.max(lo) as usize)
}
```

The `.max(lo)` is the panic-safety net: on an error/recovery path a rule may
capture `lo = cur.start` and then consume nothing (a bad token, or EOF), leaving
`prev_end < lo`, which would make `Span::new` panic. Clamping `end = max(prev_end,
lo)` yields a zero-width span instead. For well-formed nodes `prev_end >= lo`, so
it is a no-op and spans match the byte ranges ANTLR produced.

Text is read back from spans with `slice_tok(token)` and `slice_span(span)`,
both of which subtract `base` to index the trimmed buffer.

## 6. Declarations

`parse_root` implements `ast : decl* EOF`:

```rust
while !self.at(Eof) {
    let mark = self.ntok;
    match self.parse_decl() {
        Some(decl) => decls.push(decl),
        None => { self.error_at(…); self.recover_to_decl(); }
    }
    if self.ntok == mark { self.bump(); }   // progress guard
}
```

`parse_decl` dispatches on `cur` to one rule per declaration keyword:

| Keyword  | Rule                | AST node      | Shape |
|----------|---------------------|---------------|-------|
| `enum`   | `parse_enum_decl`   | `EnumDecl`    | `enum Name { ident, … }` |
| `struct` | `parse_struct_decl` | `StructDecl`  | `struct Name { field: Ty, … }` |
| `cell`   | `parse_cell_decl`   | `CellDecl`    | `cell Name(args) scope` |
| `fn`     | `parse_fn_decl`     | `FnDecl`      | `fn Name(args) (-> Ty)? scope` |
| `const`  | `parse_const_decl`  | `ConstantDecl`| `const Name: Ty = expr;` |
| `mod`    | `parse_mod_decl`    | `ModDecl`     | `mod Name;` |

Argument declarations (`argDecl : ident COLON tySpec`) and enum variants /
struct fields are comma-separated lists parsed by the shared `separated_list`
helper (§8).

## 7. Type specifications

`parse_ty_spec` parses `tySpec : ident | LBRACK tySpec RBRACK | LPAREN tySpecList
RPAREN` into a `TySpec { kind, span }` where:

```rust
enum TySpecKind { Ident(Ident), Seq(Box<TySpec>), Tuple(Vec<TySpec>) }
```

- `Float` → `Ident`
- `[T]` → `Seq` (a sequence/list type)
- `(A, B)` → `Tuple`; `()` → empty `Tuple`; `(A,)` and `(A)` → 1-element `Tuple`

The tuple element list goes through `separated_list`, so **an empty `()` and a
trailing comma both parse**. The empty tuple type is the **unit type**: the
later TySpec→`Ty` lowering (`ty_from_spec` in `compile.rs`) maps an empty
`Tuple` to `Ty::Nil` — the same type as the `()` *value* (`Expr::Nil`) — so `()`
is a real, usable type rather than an unhandled edge case. `parse_ty_spec` is
also guarded by `enter_depth`/`exit_depth` because `[`/`(` nest recursively.

## 8. Comma-separated lists: `separated_list`

Almost every comma list in the grammar is `item (',' item)* ','?` — zero or more
items with an **optional trailing comma**. That policy lives in exactly one
place:

```rust
fn separated_list<T>(&mut self, close: TokenKind,
                     mut parse_item: impl FnMut(&mut Self) -> T) -> Vec<T> {
    let mut items = Vec::new();
    while !self.at(close) && !self.at(TokenKind::Eof) {
        items.push(parse_item(self));
        if !self.eat(TokenKind::Comma) { break; }
    }
    items
}
```

Call sites: argument declarations, enum variants, struct fields, tuple-type
elements, and keyword arguments. Centralizing the loop is what keeps
trailing-comma handling and the termination guarantee from drifting between them
(that drift previously let an empty tuple type slip through silently).

**Termination.** Every iteration that does not `break` consumes at least the
separator comma, so the loop runs at most once per remaining comma — a
non-consuming `parse_item` cannot spin.

> **Not every list uses it.** Two constructs are *comma-terminated*, not
> comma-separated — a comma after **every** element is mandatory — so they keep
> their own loops:
>
> - **Tuple expressions** (`tupleExpr : LPAREN tupleExprList RPAREN` where
>   `tupleExprList : expr COMMA (expr COMMA)*`). The mandatory comma is what
>   disambiguates a parenthesized group `(a)` (no node — the inner expression's
>   own span is kept) from a one-tuple `(a,)`. `(a, b)` is therefore a **syntax
>   error**; the tuple is `(a, b,)`.
> - **Match arms** (`matchArm : identPath FAT_ARROW expr COMMA`). The comma is
>   part of each arm, and `matchArms : matchArm+` requires at least one arm, so
>   `match k {}` is a syntax error.

## 9. Expressions: the Pratt parser

`parse_expr(min_bp)` is the heart of the parser. It implements precedence
climbing:

```rust
fn parse_expr(&mut self, min_bp: u8) -> Expr {
    let lhs_start = self.cur.start;
    let mut lhs = self.parse_prefix();          // a prefix op or a primary
    loop {
        let k = self.cur.kind;
        // suffix cluster — binds tighter than any binary operator
        if SUFFIX_BP >= min_bp && k is one of { . [ ! as } {
            lhs = self.parse_suffix(lhs, lhs_start);
            continue;
        }
        // infix binary / comparison
        if let Some((op, l_bp, r_bp)) = infix_op(k) {
            if l_bp < min_bp { break; }
            self.bump();
            let rhs = self.parse_expr(r_bp);
            lhs = self.make_infix(op, lhs, rhs, lhs_start);
            continue;
        }
        break;
    }
    lhs
}
```

`min_bp` is the minimum left binding power an operator must have to bind at this
point. A top-level call passes `min_bp == 0` (anything binds).

### 9.1 Binding powers

`infix_op(kind)` is the single source of truth for the infix operator set: it
returns `(InfixOp, left_bp, right_bp)` or `None`. The `InfixOp` carries the AST
operator identity (`Bin(BinOp)` or `Cmp(ComparisonOp)`) alongside its
precedence, so the two cannot drift apart.

| Operators                         | (left, right) | Notes |
|-----------------------------------|---------------|-------|
| `== != >= > <= <`  (comparison)   | `(1, 2)`      | lowest |
| `+ -`  (additive)                 | `(3, 4)`      | |
| `* / %`  (multiplicative)         | `(5, 6)`      | |
| suffix cluster `. [] ! as`        | `SUFFIX_BP = 7` | tighter than any binary op |
| prefix unary operand `! -`        | `PREFIX_BP = 9` | see §9.4 |

Because each operator's right binding power is one greater than its left, all
binary operators are **left-associative**: `a - b - c` parses as `(a - b) - c`
(when the loop returns from parsing `b` at `r_bp = 4`, the next `-` has
`l_bp = 3 < 4`, so it does not bind to the right and instead folds at the outer
level).

### 9.2 Prefixes, primaries, and suffixes

- **`parse_prefix`** handles the prefix unary operators `!` (logical not) and `-`
  (negate). It parses its operand with `min_bp = PREFIX_BP`. Otherwise it falls
  through to `parse_primary`.
- **`parse_primary`** parses an atomic operand (the Pratt "null denotation"):
  - `(` → `parse_paren`: `()` is `nil`; `(e)` is a parenthesized group
    (unwrapped, no node); `(e, …,)` is a tuple.
  - `[` → `parse_seq_nil`: only the empty `[]` is an expression (the
    sequence-nil literal). A non-empty `[…]` is *not* an expression.
  - `if` / `match` / `{` → the block-form primaries `IfExpr`, `MatchExpr`,
    `Scope`. These are full expressions, so the Pratt loop can still attach
    trailing operators to them.
  - `#name` → `parse_annotated_primary` (an annotation, which may precede a
    call, an `if`, or a `{` block — nothing else).
  - identifier → an `identPath` (`a::b::c`); becomes a `Call` if followed by
    `(`, otherwise an `IdentPath`.
  - integer / string / `true` / `false` → the corresponding literal.
- **`parse_suffix`** applies one postfix operator to the accumulated `lhs`:

  | Syntax     | AST node            | Meaning |
  |------------|---------------------|---------|
  | `e.field`  | `FieldAccessExpr`   | struct field access |
  | `e.0`      | `IndexFieldAccessExpr` | tuple element access |
  | `e[i]`     | `IndexExpr`         | sequence index |
  | `e!`       | `EmitExpr`          | emit |
  | `e as Ty`  | `CastExpr`          | cast |

  The loop applies suffixes left to right, so `r.1[0]` is `(r.1)[0]` =
  `Index(IndexFieldAccess(r, 1), 0)`.

### 9.3 Integers vs floats

The lexer produces `IntLit` and `Dot` separately; `parse_int_or_float` assembles
floats from `IntLit DOT IntLit?`:

- `100.` → `FloatLiteral` (100.0) — a trailing dot with no fractional digits is
  still a float.
- `1.5` → `FloatLiteral`.
- `1.foo` → **field access on an integer**, `FieldAccess(IntLiteral(1), foo)`.
  When the token after the `.` is an identifier, the `.` is a field-access
  suffix, not a fractional part, so it is left for the Pratt suffix loop. (Ints
  have no fields, so this is semantically meaningless and will fail later type
  checking — but it parses uniformly, like every other primary's `.field`.)
- `1.0.2` → `IndexFieldAccess(FloatLiteral(1.0), 2)`: the first `.0` completes
  the float, the second `.2` is a tuple-index suffix.

A literal's value is parsed from the raw source slice and defaults to `0` /
`0.0` on overflow, matching the old behavior.

### 9.4 The `lhs_start` span subtlety, and prefix vs suffix

`parse_expr` records `lhs_start = self.cur.start` *before* parsing the prefix,
and operator nodes span from there rather than from `lhs.span().start()`. The
reason: a parenthesized operand is unwrapped to its inner node, whose span
excludes the `(`; but ANTLR (and the rest of the compiler) expects the enclosing
operator to span from the `(`. Using the lexical start of the first token gets
this right.

A consequence of `PREFIX_BP (9) > SUFFIX_BP (7)` is that a prefix operator's
operand is a bare primary — suffixes do **not** bind inside it. So `-a.b` parses
as `(-a).b` (`FieldAccess(Neg(a), b)`), and `-a!` as `(-a)!`. The suffix cluster
binds tighter than the binary operators, but a prefix operator grabs only its
immediate primary and the suffix then applies to the whole unary expression.

## 10. Scopes, statements, and tails

`parse_scope` = optional `#annotation` then `parse_unannotated_scope`. An
`unannotatedScope` is `LBRACE statements expr? RBRACE` — a brace-delimited list
of statements optionally ending in a *tail expression* (the scope's value).

The statement loop dispatches on `cur`:

- `let name = expr ;` → `Statement::LetBinding`
- `for v in expr scope` → `Statement::ForLoop`
- anything else → an expression statement. The expression is parsed, then:
  - followed by `;` → `Statement::Expr { semicolon: true }`
  - followed by `}` or EOF → it becomes the scope's **tail** (and the loop
    breaks)
  - a block-form expression (`if`/`match`/`{…}`) followed by more tokens → a
    `Statement::Expr { semicolon: false }` (a bare block used as a statement)
  - otherwise → record a "expected ';'" error and treat it as a statement

Because the loop already routes a trailing expression into the tail, and only
pushes a `semicolon: false` statement when more tokens follow it, a
`semicolon: false` statement is never the last element — so no separate
"tail fixup" is needed after the loop. (The `Scope.span` covers only the braces;
an annotation, if present, has its own span and is excluded.)

## 11. The `parse_cell` contract

`parse_cell_entry` parses a single cell invocation for the language server,
which reads the target cell's name and literal arguments from it. The contract
is *exactly one call expression followed by EOF*:

```rust
let Expr::Call(call) = self.parse_expr(0) else { error; return None };
if !self.at(Eof) { error; return None };
Some(call)
```

So `top(1, 2)` is accepted, while trailing garbage (`top() junk`) and suffixed
calls (`top()!`, `top().x`, `top()[0]` — whose root node is an `Emit` /
`FieldAccess` / `Index`, not a `Call`) are all rejected. This is distinct from
`format_cell_input` in `parse.rs`, which wraps a snippet into a whole `cell { … }`
program for `parse_ast`; the two are not interchangeable.

## 12. Error handling and recovery

The parser's defining property is that it never gives up and never panics. Four
mechanisms cooperate:

### 12.1 Accumulate, don't abort

Errors are pushed onto `self.errors` (as `ParseError { span, message }`) and the
parse continues. `finish_errors` returns the collected list, synthesizing a
generic "syntax error" spanning the whole input if — somehow — the parse failed
but recorded nothing, so callers always get at least one diagnostic.

### 12.2 Same-offset de-duplication

`error_at` suppresses only an **exact duplicate of the immediately preceding
diagnostic** (same start offset *and* same message) — the cascade a retried
`expect` produces while the cursor is stuck on a bad token:

```rust
if let Some(last) = self.errors.last()
    && last.span.start() == span.start()
    && last.message == message
{ return; }
```

Distinct diagnostics at the same offset are *kept* — e.g. a token that is
simultaneously "not an expression" and "not the expected `)`" yields both
messages — so independent problems are not collapsed away.

### 12.3 Progress guards

Every list/statement loop is structured so it cannot spin on input that consumes
nothing. The canonical form records `ntok` (the monotonic consumed-token count)
at the top of the iteration and force-advances if nothing was consumed:

```rust
let mark = self.ntok;
// … parse one item, which may fail and consume nothing …
if self.ntok == mark { self.bump(); }   // unstick
```

Combined with `expect`'s non-consuming synthetic token, this guarantees forward
progress without losing the ability to report an error at the stuck token.

### 12.4 Synchronizing recovery

At the top level, `recover_to_decl` skips tokens until the next declaration
keyword (`enum`/`struct`/`cell`/`fn`/`const`/`mod`) or EOF, so one malformed
declaration doesn't poison the rest of the file.

## 13. Relationship to `Argon.g4`

`core/compiler/grammar/Argon.g4` is a **descriptive reference grammar**, not an
input to any build step (nothing generates code from it any more). It documents
the intended language and should be kept in sync when the parser's accepted
language changes. Expression precedence and associativity in the Pratt table
were validated against the precedence ANTLR's generated `expr` rule produced.

## 14. Performance

The parser is designed to be fast and allocation-light:

- **O(n) single pass.** Each byte is examined a constant number of times; there
  is no backtracking (the two-token window suffices).
- **Zero-copy tokens.** No allocation for identifier or string text; values are
  borrowed `&str` slices. The only allocations are the AST nodes themselves
  (and `Box`es for the recursive `Expr` variants) plus error-message strings.
- **Jump-table keyword lookup** with no hash-map setup.

`parser::tests::parser_throughput` (ignored by default) reports lex+parse
throughput on a synthetic program:

```bash
cargo test -p compiler --release -- --ignored --nocapture parser_throughput
```

## 15. Testing

Parser tests live in `mod.rs` under `#[cfg(test)] mod tests`:

- **`accepts_valid_constructs` / `rejects_invalid_constructs`** — small accept /
  reject snippets (each wrapped in `cell __t__() { … }`).
- **`literal_values_and_spans`** — asserts concrete AST variants *and* that each
  node's span re-slices to exactly the source text it covers (spans must be
  byte-exact).
- **`int_dot_field_access_parses`, `tuple_types_parse`,
  `parse_cell_requires_a_single_invocation`,
  `distinct_diagnostics_at_same_offset_are_kept`,
  `non_ascii_char_is_one_full_width_error_token`** — focused regression tests for
  individual behaviors.
- **`corpus_parses`** — every grammar-valid `.ar` file under `examples/`,
  `pdks/`, and the compiler's `src/std/` must parse without error.

End-to-end coverage (parse → compile → solve) lives in `core/compiler/src/lib.rs`
as the `argon_*` and `stress_*_smoke` tests. There is no longer a differential
test against ANTLR (it has been removed), so behavioral changes should be locked
in with a focused `parser::tests` case.

Verify any change with:

```bash
cargo test -p compiler --lib
cargo clippy -p compiler --all-targets
cargo fmt -p compiler -- --check
```

## 16. Gotchas and invariants

- **Spans must be byte-exact** and must not invert (`end >= start`). Prefer
  `finish_span(lo)` over a raw `Span::new(lo, prev_end)` on any path that might
  consume nothing.
- **Token boundaries are UTF-8 char boundaries.** The lexer guarantees this,
  including on the `Error` recovery path; relying on it is what makes
  `slice_span` infallible.
- **`offset_base`** is added by the lexer so spans index the *original* input;
  `slice_*` subtracts `base` to index the trimmed buffer. Don't mix the two
  coordinate systems.
- **The depth guard is required.** Removing `enter_depth`/`exit_depth` from
  `parse_expr`, `parse_unannotated_scope`, or `parse_ty_spec` reintroduces a
  stack-overflow crash on deeply nested input.
- **`parse_cell` ≠ `parse_ast`.** `parse_cell` wants a bare invocation;
  `format_cell_input` produces a whole-program wrapper for `parse_ast`.
